운영 방식

요청이 들어오면 항상 같은 기준으로 설계합니다.

앞으로 들어올 디자인 요청을 이 허브에 누적하고, 각 요청마다 근거·시안·결정사항을 남기는 구조입니다.

  1. 01Brief Intake

    목표, 사용자, 화면 종류, 톤, 참고 서비스를 짧게 수집합니다.

  2. 02Spec by Gemini

    Gemini 3.1 Pro로 색상, 타입, 밀도, 레이아웃 기준을 정합니다.

  3. 03Stitch Prompt

    Stitch가 바로 이해할 수 있는 구체적인 프롬프트로 변환합니다.

  4. 043 Design Takes

    보수적·강력 추천·실험적 방향을 비교 가능한 형태로 만듭니다.

  5. 05Review & Ship

    선택안을 다듬고 HTML/React/Next.js 등으로 배포합니다.

첫 테스트 세트

같은 목적, 완전히 다른 3가지 디자인 언어

단순 색상 변경이 아니라 정보 밀도, 검토 방식, 인터랙션 모델 자체를 다르게 잡았습니다.

앞으로의 요청 방식

디자인 요청은 짧아도 됩니다. 대신 결정 기준은 남깁니다.

아래 포맷으로 요청하면 Gemini/Stitch 워크플로에 바로 넣을 수 있습니다.

Request template
목표: 어떤 화면/사이트를 만들고 싶은지
사용자: 누가 쓰는지
톤: 전문적 / 따뜻함 / 실험적 / 고급스러움
참고: 닮았으면 하는 서비스나 싫은 예시
필수 요소: 반드시 들어가야 하는 콘텐츠
출력: 3개 시안 / HTML / Next.js / PPT 등

GitHub 운영

공개 저장소 + GitHub Pages로 운영

이 사이트는 전용 공개 저장소에서 관리하고, GitHub Pages로 배포하기 좋게 순수 HTML/CSS/JS로 구성했습니다.

요청 누적

requests/에 브리프와 결정사항을 보관합니다.

시안 비교

매 요청마다 최소 3개 방향을 비교합니다.

선택 후 구현

선택된 방향만 코드로 강화해 산출물 품질을 유지합니다.
